de l'ensemble du site

Après la correction de /login, revue complète des pages pour vérifier qu'aucune autre n'est cassée sur smartphone. Verdict : le login n'était pas un cas isolé — le même anti-pattern desktop-first se répète sur une dizaine de pages. Voici le diagnostic priorisé et les remèdes.

Synthèse

Les fondations sont saines (sidebar en overlay + burger, onglets → <select>, inputs 16 px, DataList qui bascule en cartes, graphes en ResponsiveContainer). Mais ces protections ne couvrent pas les nombreux layouts « 2 colonnes » et grilles à largeur fixe écrits en inline, qui n'ont aucun point de rupture mobile.

Barème. P0 la page déborde horizontalement ou le contenu devient inutilisable sous ~375 px. P1 ça tient mais c'est tassé / illisible / grilles non adaptées. P2 finition (doigt, lisibilité). Sévérités issues d'une analyse statique — à confirmer sur appareil réel lors des correctifs.

Le constat systémique : « le pattern login » répété

La page login était cassée parce qu'elle appliquait un layout 2 colonnes desktop sans @media. Exactement le même schéma se retrouve, en inline, sur de nombreuses vues maître-détail et grilles.

La signature du problème

Un conteneur display:flex avec un panneau à largeur fixe (width:260, flex:'0 0 280px') ou des ratios (flex:3 / flex:1.8), sans aucun point de rupture pour empiler en mobile. Les styles étant inline, ils ne peuvent pas porter de media query — et la seule règle globale de secours (main > div > div[display:flex] { flex-wrap:wrap }) ne cible que les enfants directs et ne corrige ni les largeurs fixes, ni les grilles, ni les hauteurs calc(100vh − N).

→ Résultat sur 375 px : une liste latérale de 280 px laisse ~75 px au détail, ou la grille de 7 colonnes fait 750 px de large → scroll horizontal.

C'est une dette structurelle, pas une série de bugs isolés : le bon réflexe est un petit jeu d'utilitaires CSS (cf. § Remèdes) appliqués à tous ces conteneurs, plutôt que dix correctifs séparés.

Remèdes — par pattern, pas page par page

Comme la cause est unique (layouts inline sans breakpoint), la bonne approche est un petit jeu d'utilitaires CSS dans globals.css, posés en className sur les conteneurs concernés. Quelques classes corrigent la majorité des constats.

Remède 1 — le plus impactant (couvre tous les P0 « maître-détail »)

Empiler les layouts 2 colonnes sous 767 px

Une classe .split sur le conteneur flex + .split__aside sur la liste latérale. En mobile : passage en colonne et liste pleine largeur (idéalement avec un retour « ← liste / détail » ou un <select>, comme la sous-nav).

/* globals.css */
.split { display:flex; gap:var(--gap); }
.split__aside { width:280px; flex-shrink:0; }
@media (max-width:767px){
  .split { flex-direction:column; }
  .split__aside { width:100%; }     /* la liste passe pleine largeur */
}

Cible : Fournisseurs, Agents, Équipe, Bibliothèque, Commissions détail, Factures, Tâches liste, Demandes, Comptes rendus.

Remède 2 — grilles

Reflow des grilles à colonnes fixes

  • Grilles « cartes » (repeat(3,1fr)) → repeat(auto-fill,minmax(220px,1fr)) (déjà utilisé, et bon, dans Profil délégations).
  • Grilles « tableau » à largeurs fixes (Journal, Biens, Demandes) → soit basculer sur DataList (qui sait déjà se transformer en cartes), soit une classe .grid-reflow qui passe à 1 colonne < 767 px.
Remède 3 — défilement assumé

Kanban & calendriers : scroll horizontal encadré

Plutôt qu'un débordement « invisible », un conteneur overflow-x:auto explicite avec des colonnes à largeur minimale (le Kanban se scrolle horizontalement, comportement attendu). Pour le calendrier mensuel, garder 7 colonnes en scroll-x, ou proposer une vue liste/agenda en mobile.

Remède 4 — finition

Tactile, hauteurs, barres KPI

  • Boutons d'action des tableaux : viser min-height:44px (ou espacement ≥ 8 px) sur mobile.
  • height:calc(100vh − N) → 100dvh + env(safe-area-inset-bottom) (comme sur le login).
  • Barres KPI : flex-wrap:wrap explicite sur chaque conteneur de KpiCard (ne pas dépendre de la règle globale superficielle).

Anti-récidive

Pour éviter de recréer ce type de page : (1) une convention — tout layout multi-colonnes passe par .split / .grid-reflow plutôt que des largeurs inline ; (2) un réflexe de test à 375 px (DevTools) avant chaque livraison de page ; (3) privilégier DataList pour toute donnée tabulaire (bascule cartes automatique).

Plan d'action proposé

Par ordre de priorité et de rapport bénéfice/effort. Un seul lot de remèdes corrige l'essentiel.

  1. Lot 1 · P0 Créer les utilitaires .split / .grid-reflow dans globals.css et les poser sur les 8 vues maître-détail / grilles fixes. Le plus gros gain.
  2. Lot 2 · P1 Reflow des grilles restantes (Commissions, Profil, Dashboard, Calendrier) + barres KPI flex-wrap.
  3. Lot 3 · P1 Kanban & TaskCalendar : scroll-x encadré (ou empilement) + 100dvh.
  4. Lot 4 · P2 Cibles tactiles ≥ 44 px sur les boutons d'action, polices du calendrier, formulaires multi-colonnes.

Suggestion : commencer par un écran emblématique (p. ex. Finances › Fournisseurs ou RH › Agents), valider l'approche .split en réel, puis dérouler sur les autres — comme on a fait pour le login.
