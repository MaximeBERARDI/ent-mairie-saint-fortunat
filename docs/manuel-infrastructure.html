1.Vue d'ensemble — l'analogie de la boutique

Avant tout : un schéma simple pour visualiser qui fait quoi.

📚 L'analogie : Imagine que tu ouvres une boutique en ligne.
  • GitHub, c'est le coffre-fort où sont rangés les plans de la boutique (le code).
  • Vercel, c'est l'usine qui prend les plans et construit la boutique à chaque modification.
  • Supabase, c'est l'entrepôt où sont stockées toutes les informations (clients, commandes, stock).

Le client (toi, les agents) entre dans la boutique via son navigateur. Il ne voit que la boutique finie, mais derrière elle se connecte à l'entrepôt pour aller chercher / écrire des infos.

2.Les 3 services externes

GitHub stocke, Vercel publie, Supabase mémorise. Chacun a un rôle précis.

🐙

GitHub

Le coffre-fort à code

C'est quoi : un site qui stocke des "dépôts" (repositories) de code. Le tien s'appelle ent-mairie-saint-fortunat.

À quoi ça sert : garder une copie de référence du code, l'historique de toutes les modifications, et permettre la collaboration.

▲

Vercel

L'usine à site web

C'est quoi : un service qui prend le code de GitHub et le transforme en site web accessible publiquement.

À quoi ça sert : à chaque fois qu'on push du code sur GitHub, Vercel reconstruit et déploie automatiquement l'app.

⚡

Supabase

La base de données

C'est quoi : une base de données PostgreSQL dans le cloud, accessible via internet.

À quoi ça sert : stocker toutes les données métier (comptes, tâches, factures, etc.) de manière persistante et partagée.

Pourquoi 3 services différents ?

On aurait pu tout mettre sur un seul serveur (code + site + base de données), mais c'est plus fragile et plus compliqué à maintenir. Avec ces 3 services spécialisés :

  • Chacun fait son métier mieux que personne (GitHub est imbattable sur le stockage de code, Vercel sur l'hébergement Next.js, Supabase sur les DB Postgres).
  • Si l'un tombe, les autres tiennent. Si Vercel a un pépin, le code reste sur GitHub, les données restent sur Supabase.
  • Les 3 sont gratuits jusqu'à un certain volume (suffisant pour une commune).

Tes rôles sur chaque service

ServiceCompte au nom deCe que tu fais dessus
GitHub MaximeBERARDI Tu reçois et valides les modifications. Tu génères des PATs (jetons temporaires) pour me laisser pousser du code.
Vercel maximeberardi Tu configures les variables d'environnement (DATABASE_URL, AUTH_SECRET, etc.), tu surveilles les déploiements et les logs.
Supabase (ton compte) Tu exécutes les commandes SQL (via le SQL Editor), tu gères le mot de passe de la base, tu peux consulter les tables.

3.Les technologies dans l'app

Ce qui se passe à l'intérieur du code — vu de très haut.

▲

Next.js

Framework de l'app

C'est quoi : le squelette de l'application, qui définit comment construire des pages web.

Tu vois : chaque URL (/dashboard, /taches, /finances) correspond à un fichier page.tsx.

▣

Prisma

Le traducteur DB

C'est quoi : un outil qui permet au code TypeScript de "parler" à la base PostgreSQL.

Tu vois : db.user.findUnique(...) dans le code = "SELECT * FROM users WHERE..." en SQL.

🔐

NextAuth (Auth.js)

Le gardien

C'est quoi : une bibliothèque qui gère tout ce qui touche à la connexion (login, sessions, cookies).

Tu vois : la page /login, l'API /api/auth/*, la déconnexion, les cookies sécurisés.

Le langage : TypeScript

Le code est écrit en TypeScript (.ts ou .tsx pour les fichiers React). C'est du JavaScript avec en plus des "types" qui empêchent les erreurs idiotes (ex : passer un nombre là où on attend du texte). C'est l'équivalent d'un correcteur orthographique pour le code.

Le stockage des mots de passe : bcrypt

Quand tu saisis ton mot de passe, il n'est jamais stocké en clair. La bibliothèque bcrypt transforme ton mot de passe en une chaîne unique illisible (un "hash"), du genre :

$2a$10$xN4QkW.zhO5VFv5ZmI8Pue.zJxa9HtT5jvU7BqM8fEK

Quand tu retournes te logger, bcrypt re-calcule le hash de ce que tu tapes et compare. C'est impossible de retrouver le mot de passe d'origine à partir du hash. Même nous, en regardant la table users dans Supabase, on ne voit que des hashs.

4.Le cycle de vie d'une modification

Ce qui se passe entre "je veux X" et "X est en ligne".

┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ 1. TOI │ ─▶ │ 2. CLAUDE│ ─▶ │ 3. GITHUB│ ─▶ │ 4. VERCEL│ ─▶ │ 5. TOI │ │ │ │ │ │ │ │ │ │ │ │ "Je veux │ │ écrit le │ │ stocke │ │ rebuild │ │ teste │ │ X" │ │ code │ │ le code │ │ +deploy │ │ X │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ │ ◀────────── feedback ─────────── │ ▼ ▼ modifications nouvelle complémentaires demande

Concrètement, étape par étape

  1. Toi : tu m'écris une demande ("Ajoute un bouton X", "Corrige le bug Y").
  2. Moi : je modifie les bons fichiers (Edit / Write).
  3. Moi : je teste le build local pour vérifier que rien ne casse (npm run build).
  4. Moi : je crée un commit = un instantané daté et nommé des modifications.
  5. Moi : je te demande un PAT (Personal Access Token) pour pouvoir pousser sur GitHub.
  6. Toi : tu génères un PAT et tu me le donnes.
  7. Moi : je pousse le commit sur GitHub (git push).
  8. Toi : tu révoques le PAT (sécurité — il a servi, plus besoin).
  9. GitHub : reçoit le commit, notifie automatiquement Vercel.
  10. Vercel : récupère le code, le compile, et déploie une nouvelle version.
  11. Toi : tu testes sur l'URL Vercel.
  12. Toi : tu me dis si ça marche ou pas → on enchaîne.
💡 Pourquoi les PATs ?
Je tourne dans un environnement isolé (chez Anthropic). Je n'ai pas tes identifiants GitHub permanents — c'est une mesure de sécurité. Tu me fournis un jeton temporaire à chaque besoin, je l'utilise une fois, tu le supprimes. Comme une clé d'hôtel : valide pour une nuit, jetée à la sortie.

5.L'authentification en détail

Ce qui se passe entre "je tape mon mot de passe" et "j'arrive sur le dashboard".

Le scénario "je me connecte"

  1. Tu tapes berardi.maxime@gmail.com + ton mot de passe sur la page /login.
  2. Le navigateur envoie ces 2 informations en HTTPS à l'URL /api/auth/callback/credentials sur Vercel.
  3. Vercel exécute le code de NextAuth dans une fonction serveur (une "serverless function").
  4. NextAuth appelle Prisma qui se connecte à Supabase et fait : SELECT * FROM users WHERE email = '…'.
  5. Supabase renvoie la ligne avec le hashedPassword (hash bcrypt).
  6. NextAuth utilise bcrypt pour comparer le mot de passe que tu as tapé avec le hash. → OK ou KO.
  7. Si OK : NextAuth génère un JWT (un jeton signé numériquement avec AUTH_SECRET) qui contient ton identité.
  8. Ce JWT est placé dans un cookie nommé __Secure-authjs.session-token, envoyé à ton navigateur.
  9. Le navigateur stocke le cookie et te redirige vers /dashboard.

Le scénario "je clique sur Tâches" (déjà connecté)

  1. Tu cliques sur /taches.
  2. Ton navigateur envoie automatiquement le cookie __Secure-authjs.session-token avec la requête.
  3. Sur Vercel, le middleware intercepte la requête avant d'afficher la page.
  4. Le middleware décode le JWT du cookie. Il vérifie qu'il est valide (signature OK, non expiré).
  5. Si OK → la page /taches s'affiche. Sinon → redirection vers /login.

Pourquoi un cookie + JWT, et pas juste "je stocke la session sur le serveur" ?

Avec un JWT signé, Vercel n'a pas besoin de redemander à Supabase à chaque clic "est-ce que cette personne est connectée ?". Le cookie contient tout, signé numériquement — c'est instantané et économise des requêtes à la base. Tant que la signature est valide (vérifiée avec AUTH_SECRET), la session est de confiance.

⚠️ Si jamais AUTH_SECRET fuite : n'importe qui pourrait forger des JWT valides et se faire passer pour n'importe quel utilisateur. C'est pour ça qu'il est marqué "Sensitive" dans Vercel — il ne doit JAMAIS apparaître dans le code ni dans des logs.

6.Les variables d'environnement

Ce que Vercel doit savoir, sans que ça soit écrit dans le code source.

Le code public sur GitHub ne doit jamais contenir de mots de passe, de clés d'API, ou d'URL de base de données. Ces informations sont mises dans des "variables d'environnement" configurées dans Vercel.

Les 4 variables actuelles

VariableÀ quoi ça sertSi elle est cassée…
DATABASE_URL L'adresse de Supabase pour les requêtes runtime (lecture, écriture). L'app plante au login : "DATABASE_URL not found" ou "tenant not found".
DIRECT_URL L'adresse Supabase pour les migrations Prisma (création / modification de tables). Migrations impossibles. Le runtime continue normalement.
AUTH_SECRET La clé secrète qui signe les JWT de session NextAuth. Plus aucune session valide (faut se reconnecter). Si elle fuit : usurpation.
NEXTAUTH_URL L'URL publique de l'app (https://ent-mairie-saint-fortunat.vercel.app). Cookies posés sur un mauvais domaine → login en boucle.

Le format des URLs Supabase

Une URL de connexion à PostgreSQL a ce format :

postgresql://USER:PASSWORD@HOST:PORT/DATABASE?options

Décomposons un de tes DATABASE_URL :

postgresql://postgres.jvigyrsgrlravjwqulhb:MotDePasse@aws-0-eu-west-1.pooler.supabase.com:6543/postgres?pgbouncer=true
            └──────────┬───────────────┘  └─────┬───┘ └────────────┬──────────────────────┘ └─┬─┘ └──┬───┘
                  USER (compte Postgres)   PASSWORD       HOST (serveur Supabase)         PORT  DB
  • USER : postgres.jvigyrsgrlravjwqulhb — le préfixe postgres. + ton "project ref" (identifiant unique de ton projet).
  • PORT 6543 = "connection pooler" (rapide, optimisé pour les requêtes courtes). C'est ce que Vercel utilise en production.
  • PORT 5432 = "direct connection" (utilisé pour les migrations qui ouvrent une connexion plus longue).
💡 Caractères spéciaux dans le mot de passe :
Si le mot de passe contient des caractères comme @, :, /, ?, #, ils cassent l'URL (qui utilise ces mêmes caractères comme séparateurs). Il faut les "encoder" (par ex. @ → %40). Le plus simple : choisir un mot de passe avec uniquement des lettres et des chiffres.

7.Comment piloter au quotidien

Les gestes utiles à connaître pour gérer l'app sans paniquer.

Voir l'état des déploiements (Vercel)

Vercel → ton projet → onglet Deployments.

  • 🟢 Ready = en ligne, fonctionne.
  • 🟡 Building = en cours de compilation (1-2 min en moyenne).
  • 🔴 Error = build échoué. Clique pour voir le log d'erreur.

Le badge "Current" indique la version actuellement servie aux utilisateurs.

Voir les logs en temps réel (Vercel)

Vercel → onglet Logs (ou "Runtime Logs"). Tu y vois toutes les requêtes qui arrivent et toutes les erreurs côté serveur. C'est l'endroit pour debugger quand quelque chose ne marche pas.

Modifier une variable d'environnement (Vercel)

  1. Vercel → Project Settings → Environment Variables.
  2. Trouve la variable → 3 points "..." → Edit.
  3. Change la valeur → Save.
  4. ⚠️ Redéploie ensuite (Deployments → dernier → "..." → Redeploy), sinon le changement ne s'applique pas.

Réinitialiser le mot de passe DB Supabase

  1. Supabase → Project Settings → Database.
  2. Section "Database password" → Reset.
  3. Note le nouveau mot de passe quelque part de sûr.
  4. Vercel → modifie DATABASE_URL et DIRECT_URL avec le nouveau mot de passe.
  5. Redéploie.

Consulter / modifier des données dans la DB

Supabase → Table Editor dans le menu de gauche. Tu peux parcourir les tables (persons, users, tasks, etc.), filtrer, et éditer les lignes directement.

Pour des requêtes plus complexes : Supabase → SQL Editor → écris du SQL et clique Run.

Exemples de SQL utiles

-- Voir tous les utilisateurs et à quelle personne ils sont liés
SELECT u.email, p."fullName", p.role
FROM users u
LEFT JOIN persons p ON u."personId" = p.id;

-- Réinitialiser le mot de passe d'un utilisateur à "saintfortunat2026"
-- (le hash bcrypt de ce mot de passe — change l'email)
UPDATE users
SET "hashedPassword" = '$2b$10$X8h9c6FbF.Mr3RkD8RKLxe9rOK4uYy7sZHGXm0G8KqRqRqVxF4qPK'
WHERE email = 'berardi.maxime@gmail.com';

-- Compter le nombre de tâches par statut
SELECT status, COUNT(*) FROM tasks GROUP BY status;

Si quelque chose plante après un déploiement

  1. Vercel → Logs → cherche les lignes en rouge (Error).
  2. Si ça ne se résout pas vite → Rollback : Deployments → trouve un ancien déploiement "Ready" qui marchait → "..." → "Promote to Production" (ou "Rollback").
  3. Préviens-moi avec le message d'erreur — je corrige le code.

Sécurité — les bonnes habitudes

  • Mot de passe DB Supabase : ne le partage à personne. Ne le mets pas dans un email ou un chat. Stocke-le dans un gestionnaire (Bitwarden, 1Password).
  • AUTH_SECRET : pareil. Si tu suspectes une fuite, génère-en un nouveau (openssl rand -base64 32) et remplace dans Vercel.
  • PATs GitHub : courte durée de vie (7 jours max), révoqués dès qu'utilisés. Un PAT = pouvoir de modifier tout le code.
  • Comptes utilisateur de l'app : encourage-les à changer leur mot de passe par défaut (saintfortunat2026) à leur première connexion.

8.Glossaire

Les termes techniques qu'on utilise souvent, expliqués simplement.

API
"Application Programming Interface". Une porte d'entrée par laquelle le navigateur communique avec le serveur. Chaque URL en /api/... est une API.
Bcrypt
Algorithme qui transforme un mot de passe en une chaîne unique illisible (hash), impossible à inverser. On stocke le hash, pas le mot de passe.
Build
L'étape pendant laquelle Vercel compile le code source en une version optimisée prête à servir. Dure 1-2 min en moyenne.
Commit
Un instantané daté et nommé des modifications. Chaque commit a un identifiant unique (ex: 064d8956) et un message qui explique ce qui a changé.
Cookie
Un petit fichier que le serveur peut demander au navigateur de stocker. Le navigateur le renvoie ensuite à chaque requête. Sert pour les sessions de connexion.
Deploy
Mettre une nouvelle version en ligne. Sur Vercel, c'est automatique après chaque git push sur la branche main.
Endpoint
Une URL d'API spécifique. Ex : /api/auth/change-password est l'endpoint pour changer son mot de passe.
Frontend / Backend
Frontend = ce qui tourne dans ton navigateur (ce que tu vois). Backend = ce qui tourne sur Vercel (le serveur, invisible). Next.js fait les deux dans un seul projet.
Git / GitHub
Git est l'outil qui suit l'historique du code. GitHub est le site web qui héberge les dépôts Git.
Hash
Une empreinte numérique unique. Bcrypt transforme "monMotDePasse123" en un hash qui ne peut pas être inversé.
JWT (JSON Web Token)
Un jeton signé numériquement qui prouve l'identité de l'utilisateur. Format : 3 parties séparées par des points, encodé en base64.
Middleware
Un bout de code qui s'exécute avant chaque requête. Le nôtre vérifie que tu es connecté avant de te laisser accéder à /dashboard.
Migration (DB)
Une modification de la structure de la base de données (ajouter une colonne, créer une table). Géré par Prisma via le SQL généré.
PAT (Personal Access Token)
Un jeton GitHub temporaire qui me permet de pousser du code. Tu en génères un quand j'en ai besoin, je l'utilise une fois, tu le révoques.
Prisma
Le "traducteur" entre le code TypeScript et la base PostgreSQL. Au lieu d'écrire du SQL, on écrit db.user.findUnique(...).
Production / Preview / Development
Les 3 environnements Vercel. Production = la version vue par tous. Preview = une version temporaire pour tester. Development = ton ordi en local.
Push
Envoyer un ou plusieurs commits depuis l'ordi local vers GitHub. git push.
Repository (repo)
Un projet Git stocké sur GitHub. Le tien : MaximeBERARDI/ent-mairie-saint-fortunat.
Schema (Prisma)
Le fichier schema.prisma qui décrit toutes les tables de la DB (colonnes, types, relations).
Seed
Un script qui pré-remplit la DB avec des données initiales (utilisateurs, comptes M14, etc.). À ne lancer qu'une fois.
Server-side / Client-side
Server-side = exécuté sur Vercel (serveur). Client-side = exécuté dans le navigateur. Le mot-clé 'use client' en haut d'un fichier indique du client-side.
SQL
Le langage pour interroger une base de données. Ex : SELECT * FROM users WHERE email = '…'.
Tenant
Sur Supabase, ton "tenant" = ton projet, identifié par son "project ref" (ex: jvigyrsgrlravjwqulhb). Tous tes données vivent dans ton tenant.
